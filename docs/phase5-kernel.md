> 到这一步你是"有测量纪律的延迟工程师"，可以按兴趣选方向。

## 前置条件

- [ ] Phase 4 完成：TMA 归因能力已建立

## 5.1 内核旁路

### 路线选择（按目标分叉）

#### 路线 A：最容易上手 → io_uring（广度入门）
- 纯软件，不碰网卡驱动
- 适合先证明"我会用户态旁路"
- SQPOLL 轮询模式

#### 路线 B：真实行情接收器 → AF_XDP（场景正确）
- HFT 行情几乎全是 **UDP 组播 (Multicast)**
- io_uring SQPOLL 在极低延迟网络接收上开销较大
- AF_XDP 走 XDP/eBPF 直接旁路网络栈
- 接近 DPDK 的零拷贝 UMEM 通道
- 可用 `xdp-filter` 过滤非目标组播包

#### 路线 C：最接近真实 HFT → DPDK（最后再碰）
- 绑网卡 + 大页 + 轮询驱动
- 折腾量最大

### PCIe 3.0 限制（APU 特性）
5600G 是 PCIe 3.0（非 5600X 的 4.0），千兆/万兆网卡范围内不成瓶颈。

## 5.2 可选专项

### 5.2.1 持久化 / WAL 延迟
- 若想往交易系统全栈走
- 顺序写优化、fsync 延迟测量

### 5.2.2 时间同步 / PTP
- HFT 对时间戳精度的要求
- 硬件时间戳 vs 软件时间戳

### 5.2.3 迁移到 DB kernel
- order book → sort/merge 执行层
- SPSC → morsel-driven 并行
- cache 优化 → 列存 cache 布局
- TMA 归因 → 算子瓶颈分析
- Arena 分配 → 执行层内存管理
- **两条线在这里合流**
