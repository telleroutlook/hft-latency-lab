# 诚实纪律

> 这才是真正的护城河。无论走到哪一阶段，这四条永远是通过标准。

## 四条铁律

1. **测量证据优先**：任何优化都拿测量证据说有效/无效，绝不凭直觉。
2. **诚实记录失败**：每个阶段至少诚实记录一个以为有用、结果没用甚至变差的优化。
3. **分布而非均值**：全程报尾延迟分布 (p50/p99/p99.9/p99.99/max)，不用平均值骗自己。
4. **可复现复盘**：产出一篇别人能复现的复盘。

## 统计显著性检验（分场景使用）

### 日常快速试错（探索阶段）
- 连跑 3 次，看 p99.9 差值是否明显大于 run 间抖动
- 够快够用，保住试错速度

### 对外发布前的正式结算关
- Baseline 组 vs Optimization 组**各跑 10 次**
- 各收集 10 个 p99.9 数据点
- 做 Mann-Whitney U 或 Bootstrap
- **p < 0.05 才宣布统计学有效**

### 定位
- 把它当"对外发布的质检关"
- 不是"每次试错的关"
- 否则严谨会反噬迭代速度

## 一篇好的复盘长什么样

"我试过 N 件优化、M 件没用、这是统计显著性证据"的诚实复盘，在 HFT、DB、游戏引擎、网络栈社区通吃——证明的不是你会炫技，是你**懂怎么不骗自己**。

## 跨线警告

- DB 的并行是**吞吐导向**
- HFT 是**尾延迟导向**
- "无锁=更快"是 DB 本能，但在 HFT 里无锁常让 p99.9 变差
- **上无锁后必须重测尾延迟，不能只看吞吐**
